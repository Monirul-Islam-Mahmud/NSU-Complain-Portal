1. Introduction


1.1 Purpose

The goal of this document is to provide a comprehensive overview of the NSU Complain Portal. It will describe the system's purpose and characteristics, as well as the system's interfaces, what the system will do, the limitations that it must operate with, and how the system will react to external things. This document is intended for both the users and the developers of the system and will be proposed to Nabeel Mohammed Sir for its approval.

1.2 Document Conventions

The IEEE SRS template was used to style the document. We used Times at 18 and 14 for the Chapter and Topic headings, respectively, and Arial at 11 for the body for text styling.

Some abbreviations and their meaning:

1.3 Intended Audience and Reading Suggestions

This document is intended for staff, users, testers, developers and project managers

1.4 Product Scope

The software system will be designed to help students or teachers against any kind of harrassment. Any users of this system can allow to submit a form with an evidence. Thus any harrassment can be easily handled by the authoriity.More specifically, this system is designed to allow students and techers to register and login then file a complaint. The system admin will get the data of the complaint. An email will be sent to the user and the person who is accused.



2. Overall Perspective

2.1 Product Perspective

The system was designed to take the information about any kind of hassle or complain from the teachers or students

2.2 Product Functions

The major functionalities of the system are described below:

  • Students/Teachers can create accounts by scanning their NSU ID card and giving proper details
  • All the information will store in the database and will show in the Profile menu
  • The victim have to submit the necessary evidence and proper information of complain in the form.
  • All the complain will go to the system admin, an email will also be sent to the user and the accused person.
  • System admin an change any kind of information

2.3 User Classes and Characteristics

There will be three user classes in the system:

  1. System Administrators:
    • They Have full access to databases.
    • They Can create and suspend Users â€“ teachers & students.
    • They Handle all the complain form
  2. Users:
    • Register and Login.
    • Add a complain via the form.
    • Can't access the homepage without proper validation.
    • Will get an email after submitting complain.

2.4 Operating Environment

  1. Backend:
    • Hardware Platform: x64
    • Operating System: Windows, MacOS, Linux
    • Programming Language: Python 3.7
    • Web Framework: Django 2.2
    • Database: SQLite
  2. Frontend:
    • Android App:
      1. Hardware Platform: arm
      2. Operating System: Android 6.0+
      3. Programming Language: Java & Kotlin
    • Web App:
      1. Framework: Django
      2. Design: HTML, CSS, Bootstrap

2.5 Design and Implementation Constraints

  • The backend should publish APIs which should be easily integrated with the existing University platform like RDS.
  • The system needs to be designed with an existing standard web framework.

2.6 User Documentation

The system is designed to be intuitive and user friendly.

2.7 Assumptions and Dependencies

The system don't use any external API for accomplishing complicated tasks. We only used Googe Sign in API for login.

4. System Features



UML Expanded Use Case

1) Use case name: Register & Login

Actors: Faculty, Student, Staff

Type: Primary

Brief Overview: Unregistered students/faculties/staff can choose this option to regsiter account using their valid name, Scan photo of NSU ID card, NSU email address and password.

Typical Courses of Events:
Actor Actions System Response
Actor goes to the website System reponds with the Register page of the website
Actor clicks the "Register" option to sign up for an account System checks the validity of the e-mail address and creates an account
After register, user can login with the email and password. If the user credintials are valid, then they can access the homepage.
Alternative Courses of Events:

(i) The email address entered is registered before. The actor gets prompted with an error message "This e-mail exist already". The actor either chooses to use a valid email address or creates one first.

(ii) The email address or password entered is/are incorrect. An error message pops up "Your email address or pasword is/are incorrect". The actor types in the correct information or chooses "forgot password" if he/she wants to.


2) Use case name: Submitting complain

Actors: Student/ Faculty

Type: Primary

Brief Overview: Faculty or teachers can submit the complain form with an proper evidence.

Typical Courses of Events:
Actor Actions System Response
Users after login, they can complain a file. System will redirect them to the complain page.
Users after completing all the information, press the send button. System decides if it's a valid or not and then respond by updating their page.
Alternative Courses of Events:

If any information didn't properly answered via users, then no response will be send.


3) Use case name: Update users and secure the data in database

Actor: System Admin

Type: Secondary

Brief Overview: The system admin is logged into the system and can update information about the student/faculty. The admin can choose to edit, add or remove any accounts & has complete control of the database.

Typical Courses of Events:
Actor Actions System Response
Admin chooses to update the user by clicking on the specific person's information System will update the user info.
Admin clicks "update" and press "save" on the admin panel. All the changes will be saved by the system.
Admin will get the submitted complain information and can edit the info. The system edits the complain information.

4.1 Registration with NSU valid info or Google sign in

4.1.1 Description and Priority

Students can register by scanning their NSU ID & can login using their northsouth.edu email.

4.1.2 Stimulus/Response Sequences

For Registration, the user will take a picture of their NSU ID and the user will create an account with their valid northsouth.edu email, NSU id and password.

4.1.3 Functional Requirements
  • The system will ensure only North South University students can access the system and the system will use the id card data to recognize the user.

4.2 Upload Complain

4.2.1 Description and Priority

Users can upload the complain form. Then system admin can edit the form.

4.2.2 Stimulus/Response Sequences

Users must have to submit valid evidence for submitting the form.

4.2.3 Functional Requirements

Users can upload any size of picture or voice documents. Check if the file is a valid document.

5. Non-Functional Requirements

5.1 Perfomance Requirement

The system should be able to render pictures.

5.2 Safety Requirements

    Using false identification. (Fake ID)

5.3 Security Requirement

  • The system will be secured and will respect the users privacy.
  • Only the administrator has the rights to manage any changes in the system. Only the administrator has the rights to manage any changes in the stored data set.
  • Students will require username and password to log into their individual account.

5.4 Software Quality Attributes

  • The User interface of the system will be user friendly.
  • The system must be flexible. Should be easily modified.
  • Execution process must be fast and responsive.
  • The software is a web-based application and is built in Python and SQLite, so it's platform independent.

5.5 Business Rules

  • Administrator Requirement: Have a complete control to the system. They will be able to edit, add/remove any user and update the form information.
  • User Requirement: They can access to the homepage agter valid login. After successfully login, they can add a complain to the authority.

6. Other Requirements

Database :

All the data will be stored in a relational database.
